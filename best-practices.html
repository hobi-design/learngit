01 · Integrity

Commit
Structure

Commit messages explain the history of your project. Start them with an action word (like 'add' or 'fix') rather than past tense. Use a standard prefix to categorize the change, and don't mix unrelated updates in a single commit.

Imperative mood · Prefix i The imperative mood expresses a direct command: “add” not “added”. Standardizes history to match auto-generated Git outputs (e.g. “Merge pull request…”) and answers: “If applied, this commit will [subject line].” 
feat: add transformer layer to model
Past tense · Vague 
added some stuff for the ai
Type Usage Scenario
feat New feature (e.g., new data pipeline)
fix Bug fix (e.g., resolving tensor dimension error)
docs Documentation only (e.g., updating README)
chore Maintenance (e.g., updating dependencies)
refactor Code restructuring without behaviour change
02 · Stability

Branch
Isolation

Isolate all active development inside dedicated feature branches. Do not commit code directly to primary production branches to prevent integration conflicts and trunk instability.

Dedicated branch 
git switch -c feature/add-transformer
Direct commit to main 
git commit -am "quick fix on main"

03 · Hygiene

Staging
Audit

Audit all changes prior to creating a commit. Review diffs hunk-by-hunk and exclude temporary configs, environment keys, and build artifacts from tracking via .gitignore.

Targeted · Hunk-by-hunk 
git add -p model.py
Blind · Everything staged 
git add .
